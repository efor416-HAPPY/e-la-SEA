# ARA 자율 유지보수 및 자가 진단 코어 사용 설명서

본 시스템은 오프라인 우선(offline-first) 정책에 입각하여 작동하는 독립형 자율 코드 점검, AI 복구, 및 롤백 안전망 장치입니다. 인공지능 '아라(ARA)'의 건강을 스스로 돌보고, 구문 버그 및 파일 무결성을 자율 검사 및 치료하기 위해 고안되었습니다.

---

## 1. 시스템 구성 및 아키텍처

유지보수 코어는 `e:\SEA\maintenance\` 폴더 아래에 다른 소스코드와 철저히 격리되어 기동됩니다.

1. **[self_diagnostics.py](file:///e:/SEA/maintenance/self_diagnostics.py) (자가 진단)**:
   - 전체 파이썬 파일의 구문 컴파일 가능 여부를 검사합니다.
   - 루트의 `verify_integrity.ps1`을 호출해 핵심 리소스 누락 및 CDN 무결성, JS 모듈 인터페이스 상태를 체크합니다.
2. **[repair_agent.py](file:///e:/SEA/maintenance/repair_agent.py) (AI 복구 에이전트)**:
   - 로컬 Ollama AI 코어와 통신하여, 진단 보고서와 피드백 내용을 수집하고 정밀 교정된 패치 코드를 생성합니다.
3. **[manager.py](file:///e:/SEA/maintenance/manager.py) (안전 오케스트레이터)**:
   - 수정 대상 파일의 실시간 백업을 생성하고 패치를 적용합니다.
   - 패치 직후 자가 진단을 구동하여, **단 하나의 구문 오류(SyntaxError)나 무결성 검증 실패(verify_integrity fail)라도 검출되면 즉시 원본으로 롤백(Rollback)**하여 작동 파손을 원천 차단합니다.

## 2. 간헐적 기능 오작동에 대한 원인 분석 (Troubleshooting Guide)

> "왜 기능들이 어떨 때는 작동하고, 어떨 때는 작동하지 않을까요?"

이 현상은 프로그램 자체의 버그가 아니라, **로컬 하드웨어 및 외부 소프트웨어 의존성** 때문에 발생합니다. 아래의 핵심 원인들을 확인해 주십시오.

### ① 로컬 AI (Ollama) 기능 실패
- **원인**: 백엔드는 로컬 포트 `http://localhost:11434`에 활성화된 Ollama 에이전트를 호출합니다. 만약 Ollama 프로그램이 켜져 있지 않거나, 지정된 AI 모델(기본값: `gemma2:2b`)이 로컬 컴퓨터에 다운로드(`ollama pull gemma2:2b`)되어 있지 않으면 백엔드 AI 채팅 및 코드 치유 기능이 오류와 함께 차단됩니다.
- **해결책**:
  1. 윈도우 작업 표시줄 트레이에서 Ollama 아이콘이 떠 있는지 확인하십시오.
  2. 터미널에서 `ollama run gemma2:2b`를 실행해 모델 작동 상태를 확인하십시오.

### ② CAD & 3D 자동화 스크립트 실패 (Inventor, AutoCAD, ArtCAM 등)
- **원인**: 파이썬의 `win32com` 라이브러리를 통한 외부 프로그램 제어 방식은 **해당 상용 소프트웨어가 PC에 실제로 설치되어 있고, 백그라운드 COM 개체 등록이 정상적으로 되어 있어야만** 작동합니다. 또한, 프로그램이 이미 켜져 있을 때만 API 획득에 성공(`GetActiveObject`)하고, 꺼져 있을 때는 권한 문제나 실행 시간 초과로 오류가 발생할 수 있습니다.
- **해결책**:
  - 해당 스크립트를 기동하기 전에 Autodesk Inventor, AutoCAD 또는 ArtCAM 소프트웨어를 먼저 실행해 두십시오.
  - 스크립트 실행 환경의 권한(관리자 권한 여부)을 확인하십시오.

## 3. 시스템의 장점과 단점 (Pros & Cons)

### 👍 장점 (Pros)
- **절대적 작동 안전성 (Rollback Protection)**: AI가 임의로 잘못 기재한 코드가 작성되어도 컴파일 단계나 무결성 스캔에서 실패하면, 즉시 `.bak` 파일에서 원본 코드로 자동 롤백되므로 서비스가 먹통이 되는 일이 없습니다.
- **오프라인 우선 (Local-First)**: 외부 유료 클라우드 API를 구독하지 않고도 로컬 컴퓨터 내부의 자원으로 진단 및 자율 치료 루프를 완벽히 수행합니다.
- **다차원 진단**: 단순히 구문 테스트뿐만 아니라 프론트엔드-백엔드 구조의 연결 부위(인터페이스, 의존 라이브러리)까지 포괄 진단합니다.

### 👎 단점 (Cons)
- **로컬 자원 의존성**: 로컬 LLM을 기반으로 구동되므로, 사용자의 CPU/GPU 및 RAM 성능이 낮으면 복구 코드(Patch) 생성 속도가 지연되거나 타임아웃이 발생할 수 있습니다.
- **외부 런타임 수동 기동 필요**: Ollama 및 수정을 원하는 CAD 소프트웨어 등이 반드시 먼저 활성화 상태여야 관련 자동화 복구가 성사됩니다.

## 4. 수동 및 CLI 사용법

유지보수 관리자 콘솔을 터미널에서 직접 실행할 수도 있습니다:

- **시스템 자가 진단 실행**:
  ```powershell
  python e:\SEA\maintenance\manager.py --diagnose
  ```
- **피드백 기반 자율 치유 작동**:
  ```powershell
  python e:\SEA\maintenance\manager.py --repair "viewer.js 파일 내의 에러 수정 및 XSS 방어 처리 적용"
  ```
